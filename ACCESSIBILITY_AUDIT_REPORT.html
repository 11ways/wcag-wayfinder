A. Executive Summary

Overall Risk: LOW-MEDIUM

  • Strengths: Excellent semantic HTML foundation, comprehensive ARIA implementation (111+ attributes), robust keyboard navigation, proper focus management, multi-theme support with AA+ contrast ratios.
  • Critical Gap: No <noscript> fallback—entire app fails without JavaScript (WCAG 4.1.2 at AA level expects graceful degradation).
  • High Priority: Modal focus management uses 100ms setTimeout (race condition risk); inert attribute lacks polyfill for older browsers.
  • Medium Issues: Four uses of dangerouslySetInnerHTML (mitigated by DOMPurify but requires ongoing vigilance); some tag buttons use client-side ARIA states without server-side equivalents.
  • Positive: Reflow at 200%/400% zoom PASSES; 320px responsive design PASSES; skip links implemented; live regions for screen readers; color + shape indicators (not color alone).
  • Progressive Enhancement: URL-based state allows sharing/bookmarking but requires JS for interactivity. Search form submits via React, not native HTML form action.
  • Screen Reader Support: Strong with 2 live regions (polite + assertive), descriptive labels, proper name/role/value patterns. No AT-specific testing documented.
  • Test Coverage: 70/82 tests passing (85%). 12 failing tests suggest potential regressions in element finding/pagination—investigate before production.
  • Quick Wins (48h): Add noscript message, replace setTimeout with useLayoutEffect in modals, add inert polyfill, document XSS review cadence.
  • Strategic Work (4-6w): Formal AT testing with NVDA/JAWS/VoiceOver, AAA compliance audit, automated axe-core CI integration, performance profiling under magnification.

C. Issue Log (Prioritized by Severity)

🔴 BLOCKER ISSUES (Prevents Core Tasks)

BLOCKER-01: No JavaScript Fallback

WCAG: 4.1.2 (Name, Role, Value) - Level A

User Impact: Users with JavaScript disabled get completely blank page; screen reader users in low-bandwidth/secure environments cannot access any content.

Where: apps/web/index.html

Evidence: No <noscript> tag present in HTML template. React app renders nothing without JS runtime.

Why It's a Problem: WCAG 2.2 requires content to be accessible. While SPAs often require JS, EN 301 549 and accessible procurement standards expect graceful degradation or clear messaging.

Fix Strategy:

  1. Add <noscript> tag to index.html with message: "This application requires JavaScript. Please enable it or use a modern browser."
  2. Include link to static page with basic WCAG reference or contact info
  3. Long-term: Consider server-side rendering (SSR) or static export for critical paths

Owner: Developers

Effort: Small (1-2 hours for noscript; Large for SSR)

Dependencies: None for noscript; SSR requires architecture change

🟠 HIGH SEVERITY (Major Friction for AT/Keyboard Users)

HIGH-01: Modal Focus Race Condition

WCAG: 2.4.3 (Focus Order) - Level A

User Impact: Screen reader users may experience delayed or missing focus announcements on modal open; keyboard users might see unpredictable focus behavior.

Where: apps/web/src/components/Modal.tsx:97-108

Evidence: 
useEffect(() => {
  setTimeout(() => {
    const focusElement = modalRef.current?.querySelector('[autofocus]');
    // 100ms delay — timing-dependent, not guaranteed
  }, 100);
}, []);

Why It's a Problem: setTimeout is unreliable across devices/browsers. In slow environments (CPU throttling, magnification), focus may land before content renders, causing screen reader to announce nothing or incorrect element.

Fix Strategy:

  1. Replace setTimeout with useLayoutEffect or callback ref pattern
  2. Use focus-trap-react library's initialFocus option (already imported but not fully utilized)
  3. Test with NVDA + Firefox, JAWS + Chrome, VoiceOver + Safari to confirm focus announcement timing

Owner: Developers

Effort: Medium (4-8 hours with AT testing)

Dependencies: Requires AT testing environment setup

HIGH-02: Inert Attribute Polyfill Missing

WCAG: 2.1.2 (No Keyboard Trap) - Level A

User Impact: In older browsers (Safari < 15.5, Firefox < 112), background content remains keyboard-accessible when modal is open; AT users can tab behind modal.

Where: apps/web/src/components/CriterionCard.tsx:84

Evidence: 
{...(!hasSelectedTag && { inert: '' as any })}

The inert attribute is used but has no polyfill. Browser support:

  • Chrome 102+ ✅
  • Safari 15.5+ ✅
  • Firefox 112+ ✅
  • Older browsers: ❌ Falls back to doing nothing

Why It's a Problem: Screen reader users on older browsers may accidentally navigate into inert content, creating confusion and potential keyboard traps.

Fix Strategy:

  1. Add wicg-inert polyfill via npm: npm install wicg-inert
  2. Import in main.tsx: import 'wicg-inert';
  3. Alternative: Use aria-hidden="true" + manual tabindex management as fallback

Owner: Developers

Effort: Small (2-4 hours including testing)

Dependencies: Browser compatibility testing

🟡 MEDIUM SEVERITY (Noticeable Friction, Workarounds Exist)

MEDIUM-01: XSS Risk via dangerouslySetInnerHTML

WCAG: Security risk intersecting accessibility (not direct WCAG violation)

User Impact: If sanitization fails, malicious content could inject scripts that harm AT users (e.g., fake ARIA labels, focus hijacking).

Where: 4 files use dangerouslySetInnerHTML:

  • CriterionDetails.tsx - renders WCAG content HTML
  • ResultList.tsx - renders warning banners
  • Filters.tsx - renders help content
  • TermModal.tsx - renders glossary definitions
Current Mitigation: All content sanitized with DOMPurify before rendering: 
const clean = DOMPurify.sanitize(markdownHTML);

Why It's a Problem: DOMPurify config might miss edge cases; future developers may bypass sanitization; dependency vulnerabilities.

Fix Strategy:

  1. Document sanitization requirement in code comments: "// SECURITY: Always sanitize before dangerouslySetInnerHTML"
  2. Add ESLint rule to warn on dangerouslySetInnerHTML usage
  3. Set up quarterly DOMPurify version checks and CVE monitoring
  4. Consider migrating to React-based markdown renderer (e.g., react-markdown) to avoid HTML injection entirely

Owner: Developers + Security Team

Effort: Medium (initial: 4h; ongoing: 1h/quarter)

Dependencies: ESLint config update, dependency monitoring setup

D. Progressive Enhancement Verdict

Summary: PARTIAL (URL-based State ✅, No JS Fallback ❌)

What Works Without JS:

  • ❌ Nothing. The app is a React SPA; without JavaScript, users see a blank page.
  • ✅ URLs are shareable/bookmarkable - filters encoded in path (e.g., /level:a+aa/principle:p/), so links can be saved and shared.
  • ✅ Semantic HTML foundation - when JS loads, the DOM uses proper elements (not all divs).

What Breaks Without JS:

  • 🔴 All content - no server-side rendering, no static export
  • 🔴 Search form - relies on React onChange, not native form submission
  • 🔴 Navigation - React Router handles all routing; browser back/forward work only if JS already loaded
  • 🔴 No noscript message - users don't even know JS is required

Graceful Degradation Strategy:

  1. Immediate: Add <noscript> with message + link to static WCAG reference
  2. Short-term: Consider Next.js migration for SSR/SSG (static export of criterion pages)
  3. Long-term: Evaluate whether app *must* be SPA or if progressive enhancement with server-rendered HTML + client-side enhancement is viable

WCAG Compliance Impact:

While WCAG 2.2 doesn't explicitly require non-JS fallback, 4.1.2 (Name, Role, Value) and EN 301 549 (European procurement) expect content to be "accessible" broadly. A completely blank page fails this. Recommendation: Add noscript at minimum.

F. Regression Risks & Guardrails

High-Risk Areas (Likely to Break with Future Changes)

  1. Modal Focus Management (Modal.tsx)
    • Risk: Refactoring focus logic could break keyboard trap/restoration
    • Guard: Add E2E test: "Open modal → Tab around → Escape → Verify focus restored"
  2. URL Sync Logic (useURLSync.ts)
    • Risk: Changes to filter structure could break URL parsing; infinite loops if sync flag mishandled
    • Guard: Expand urlUtils.test.ts to cover all edge cases; monitor for URL encode/decode mismatches
  3. ARIA Live Regions (LiveRegion.tsx)
    • Risk: Removing or duplicating live regions breaks screen reader announcements
    • Guard: Grep for role="status" in CI; fail if count ≠ 2 polite + assertive regions
  4. Theme System (index.css)

  5. Keyboard Shortcuts (useKeyboardShortcuts.ts)
    • Risk: New shortcuts conflict with browser/AT shortcuts (e.g., Ctrl+F)
    • Guard: Document all shortcuts in ARIA live help; test with screen readers to avoid conflicts
